# 《内部测试账号加白申请SOP》

## 1. 背景与适用范围

1.1 背景  
为保障线上防刷、防欺诈策略有效性，生产环境默认对异常登录、下单、支付行为进行策略拦截。但研发测试、产品验收、风控规则验证等场景，确有需要在真实生产环境中完成闭环测试，可能被现有策略误拦。  
本SOP用于规范“内部测试账号加白”申请与审批流程，避免随意加白导致真实风险流入，保证风控策略与测试需求的平衡。

1.2 适用范围  
- 申请主体：京东内部正式员工/授权外包员工（需在HR系统有记录），严禁外部用户或亲友账号代为申请。  
- 适用场景：
  - 新产品/新功能上线前的生产验证测试；
  - 新风控策略上线前后的真实流量AB验证；
  - 线上疑难问题复现，需要真实交易链路验证。  
- 适用账号类型：
  - 已完成实名、绑定公司邮箱/工号可追溯的UID；
  - 仅限个人用户账号，不含商家店铺主体UID、对公账户等。  

## 2. 判罚逻辑

2.1 准入逻辑（允许加白的条件）  
满足以下全部条件时，可考虑加白处理：

- 业务合理性：  
  - 工单中说明明确测试目的、涉及功能模块、是否存在替代方案（如测试环境/沙箱）；  
  - 说明为何必须使用生产真实环境、不可在灰度/沙箱完成。
- 可控性：  
  - 指定具体UID（禁止使用手机号模糊描述）；  
  - 明确生效时间与截止时间（默认不超过7天，特殊项目不超过30天）；  
  - 说明预计触发的核心策略类型（如：登录防刷策略、下单反作弊策略、支付风控策略）。
- 风险可追溯：  
  - UID已进行实名核身，且实名信息与申请员工信息可对应或可追踪到责任人；  
  - 工单中保留申请人、直属主管、审批人信息。

2.2 拒绝逻辑（不予加白的情形）  
出现以下任一情况，必须拒绝加白：

- 申请用途包含明显的个人获益：刷优惠券、获取补贴、薅羊毛、参与用户活动拿奖励等；  
- 账号历史存在黑产关联记录（UID / 设备 / 支付工具 / 地址已进入黑名单）；  
- 拟加白范围模糊或过大：如请求“关闭所有风控策略”“长期加白”、请求按IP/设备整体关闭策略等；  
- 无直属主管审批或业务负责人邮件/系统确认。

2.3 加白粒度建议  

- 优先策略级加白而非账号全量加白：  
  - 尽量仅对指定风控规则（如某条防刷策略 ID）放宽，而非取消所有拦截；  
  - 如确需账号级加白，必须限定场景（登录/下单/支付中的某一环节）+时间。
- 时间维度：  
  - 默认 ≤7 天，过期系统自动失效；  
  - 长期项目需按阶段拆分，多次续期，严禁“永久加白”。

## 3. 详细操作步骤

3.1 申请人准备材料  

1. 收集必要信息：  
   - 申请人姓名、工号、部门、联系方式；  
   - 测试账号UID、绑定手机号、账号实名信息简要说明；  
   - 涉及业务线/产品模块、需测试功能说明；  
   - 预计测试时间段与频率；  
   - 预计触发的防刷/风控策略类型（如登录、下单、支付）。  
2. 在工单系统创建《内部测试账号加白申请》工单，按模板填写：  
   - 【项目名称/需求编号】  
   - 【测试场景及目的】  
   - 【业务风险评估及替代方案说明】  
   - 【账号信息（UID/手机号）】  
   - 【拟加白策略范围及时间】  
   - 【责任人及回溯方式】。

3.2 审批流程  

1. 直属主管审批：  
   - 确认测试目的真实、必要，不属于个人利益行为；  
   - 确认测试时间、范围合理；  
   - 在工单中给出“同意/不同意”意见及文字说明。  
2. 业务负责人审批（必要时）：  
   - 对于影响较大项目（跨部门/影响核心支付链路），需业务负责人邮件或系统确认。  
3. 风控审批人（风控运营/策略负责人）：  
   - 根据历史行为校验UID是否存在黑产关联；  
   - 评估加白范围是否合理，是否需要改为“策略级加白”；  
   - 最终给出：同意（附具体配置要求）/驳回（附理由）。

3.3 风控系统配置  

1. 在风控平台/反作弊系统中，定位“测试账号加白名单”配置模块；  
2. 新增记录，字段要求：  
   - UID；  
   - 生效开始时间/结束时间；  
   - 加白维度（登录/下单/支付/其他策略 ID）；  
   - 操作人、工单编号、备注（项目名+申请人）；  
3. 保存并发布策略变更，记录变更版本号；  
4. 通知申请人进行测试，并提示：  
   - 不得用于真实购物获利；  
   - 测试完成后在工单中反馈结果，有异常及时回报。

3.4 后续回收与审计  

1. 到期自动失效：  
   - 每日通过任务/报表巡检“已过期未清理的加白项”，确保系统配置自动失效正常；  
2. 提前回收：  
   - 如项目提前结束，风控可根据申请人反馈提前移除加白；  
3. 审计与抽查：  
   - 每月至少抽查一次加白账号真实交易行为，如发现存在个人获利/高风险行为，立即终止加白并上报；  
   - 对异常案例录入风控知识库，用于后续优化加白规范。

## 4. 风险提示/话术

4.1 内部风险提示  

- 严禁将加白当作“特殊通道”为他人谋利；  
- 操作人不得私下通过IM、电话接受口头加白请求，必须走工单并留痕；  
- 禁止告知申请人具体拦截规则细节（如具体策略阈值、触发条件），仅说明“防刷策略”或“风控策略”；  
- 对于历史有风险标记的UID，原则上不予加白，确有特殊情况须升级至风控负责人审批。

4.2 对内部协同方（产品/研发等）的统一话术示例  

- 同意加白类：  
  > 因本次为【XXX项目】的生产验证测试，已按流程为UID：XXXX 进行了限时（至YYYY-MM-DD）加白处理，仅放宽【登录/下单/支付】相关防刷策略，其余风控策略仍正常生效。请勿用于真实购物或非测试用途，测试完成后反馈结果，我们将统一回收配置。  

- 拒绝加白类：  
  > 根据公司风控规范，测试账号加白仅限必要的生产验证场景，且需在可控范围内执行。当前申请账号存在历史风险记录/用途说明不清晰，无法满足加白要求，建议优先使用测试环境或调整测试方案。若确有业务刚性需求，请补充详细说明并升级至业务负责人确认后重新提交。
